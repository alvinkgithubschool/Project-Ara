Section 01

// Vision & Scope

Ara is the "control room" for complex creative projects. It builds a unified project graph across game engines, DAWs, creative-coding environments, collaboration tools, and cloud storage — without forcing teams to abandon their existing tools.

Priority Deep Integrations (MVP Cycle 1)

Unity / Godot

Scene graphs, asset relationships, script dependency parsing. Unity as anchor (25+ platforms), Godot as open-source counterpart.

TouchDesigner

Network extraction, COMPs, panels, and connections. Ara's canvas UX borrows from TD's panel model.

Notion + Git

Docs, tasks, and wikis as graph nodes linked to implementation artifacts. Git for versioned artifact history.

SpacetimeDB

Real-time collaborative graph state. Self-hostable. Incremental replication and client-side cache.

// Frameworks & Cross-Platform Stack

Ara is built as a cross-platform desktop application using modern frameworks that prioritize performance, security, and native integration. The stack balances web-technology flexibility with systems-level performance for file scanning and graph operations.

Why Tauri?

Performance

~600KB final binary vs 100MB+ Electron bundles. Rust backend uses <5MB RAM at idle vs 100MB+ typical Electron footprint. Native WebView rendering, not bundled Chromium.

Security Model

Multi-process sandbox architecture. Frontend runs isolated in WebView. Explicit capability-based API access (filesystem, network, shell). No Node.js in renderer process.

Native Integration

Direct OS API access via Rust: native file dialogs, system notifications, deep links, tray icons, global shortcuts. No electron-builder workarounds.

Cross-Platform

Single codebase targets Windows (WebView2), macOS (WKWebView), Linux (WebKitGTK). Mobile roadmap: Tauri v2 supports iOS/Android via native WebViews.

Section 03

// Integration Tiers

Tier 0 File-Only Awareness

Detect projects by extension and directory markers. Classify by tool family. Represent as graph nodes with containment edges and manual user links.

Extensions: .unity, .uproject, .tscn, .toe, .als, .flp, .logicx, .pde, .glsl

Tier 1 File + Metadata Graph

Parse project files to extract structure: scene trees (Godot), COMP hierarchies (TouchDesigner), imports/dependencies (scripts, shaders). Machine-readable project graph.

Parsers: Godot .tscn, TD .toe/.tox, tree-sitter for code

Tier 2 APIs / Connectors / MCP

Live state refresh, deep links ("open this scene"), and agent-accessible actions (trigger builds, create Notion pages). MCP unifies how Ara talks to tools.

Initial: Notion API + engine build hooks + Git commands

// Scanning & Relationship Model

Section 06

// Storage Architecture

PROJECT ROOT │ ├── .ara/ │ ├── graph.db ← SQLite local project graph │ ├── config.toml ← SpacetimeDB URL, Git remotes, connectors │ └── cache/ ← Thumbnails, indexes, partial parses │ ├── Assets/ ← Unity / engine assets ├── scenes/ ← Godot .tscn files ├── shaders/ ← GLSL / HLSL ├── audio/ ← DAW sessions + samples └── docs/ ← Notion exports, markdown

SpacetimeDB

Real-time graph state engine. Tables for nodes, edges, tags, comments, workspaces, users. Client subscriptions restricted to current project. Self-hostable via Docker/systemd.

Git Integration

Expose status (dirty, branch, ahead/behind). Simple commands: commit, push, pull, tag. Optionally store graph snapshots alongside code commits for time-travel.

Section 07

// Agentic Workflows

With the project graph spanning engines, DAWs, creative code, docs, and PM tools, agents can perform higher-level cross-tool workflows.

Agent Capabilities

AgentCapabilityBoundary
DocumentationTraverse graph regions → produce summaries, diagrams, doc stubs into Notion/markdownRead-only graph + write to docs
CartographerWalk cross-tool paths (shader → TD → DAW → Notion spec)Read-only across all stores
RefactoringFlag dead assets, circular deps, stale docs → propose tasks in PM toolsRead graph + propose writes
Build RunnerTrigger engine test builds, bounce audio, generate lightmapsExplicit per-project opt-in

Capability Boundaries

Per project, users configure: which folders agents may see, which SpacetimeDB tables they can read/write, which APIs they may call, and whether they can perform writes or only propose them.
